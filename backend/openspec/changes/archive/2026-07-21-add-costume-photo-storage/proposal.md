## Why

The Costume aggregate already carries the *linking* mechanics for photos (`PhotoLinked` / `PhotoUnlinked` events, `photos: Vec<Uuid>` state, `projection_costume_photo` join, `CostumePhotoView`), but there is no way to *store, serve, or delete* the actual image bytes. ADR-009 has decided the strategy (OpenDAL as the abstraction, S3-compatible backend), but nothing is implemented. We need to make photo upload real so costume staff can attach reference images to costumes — and because costume photos are confidential (faces, persons), the storage and serving path must enforce authorization on every byte transfer, not rely on bearer-token URLs.

## What Changes

- **Introduce a new `Photo` aggregate** (category `"photo"`) tracked in SierraDB as the single source of truth for photo *lifecycle* — upload, EXIF-normalisation, variant generation status, deletion. Photo bytes remain a side-effect stored in the S3 backend, but every projection is replay-derived from events (consistent with the rest of the system's disaster-recovery story).
- **Store bytes in Garage**, an S3-compatible Rust server, run as a Docker container that is **internal-only** (no host port exposure; only the API binary reaches it). OpenDAL's S3 service is the adapter. Phase 1 (`fs` service) is skipped on purpose: simulating S3 with a filesystem would mask the real integration issues (HEAD semantics, overwrite behaviour, listing, etc.) we want to surface now.
- **Three variants per photo**: `Original` (re-encoded upright, EXIF stripped), `Thumb` (~200×200), `Medium` (~800×800). Thumbnails/medium are generated by a saga reacting to `PhotoUploaded`; the saga also re-encodes the original to strip EXIF (orientation-aware, using `kamadak-exif`) so the stored original is data-economy-clean. EXIF stripping is enforced by default; a future toggle to disable it is an explicit v2 non-goal.
- **Proxy upload and proxy download**: the API receives bytes on upload and streams bytes on download; the frontend never receives a presigned URL. Every download is authorized per-request via the existing membership policy. Garage is never reachable from the internet.
- **N:M photo ↔ costume link** with refcount-driven deletion. A photo can be linked to multiple costumes; the deletion saga only deletes the bytes when the last `PhotoUnlinked` brings the refcount to zero. The link itself remains a `Costume` event (`PhotoLinked`/`PhotoUnlinked`, unchanged); the refcount is a `COUNT(*)` over `projection_costume_photo`.
- **Content-type allowlist**: `image/jpeg`, `image/png`, `image/webp` accepted on upload; HEIC/HEIF rejected with a clear error (clients convert before upload). Size cap configurable via env (`PHOTO_MAX_SIZE_MB`, default 20).
- **Authorisation v1: derived** — a user can view costume photos in Season S if they hold any costume-department role (`costume_designer`, `wardrobe_supervisor`, `costume_assistant`) in any `active` block of Season S. This deliberately leaves a "between-blocks gap" (costumers between contracts lose access); closing that gap with an additive `SeasonCrew` aggregate is an explicitly documented v2 evolution path, captured in `design.md`.
- **Periodic orphan GC**: a background sweep reconciles `projection_photo` against Garage object listing and deletes orphans older than a configurable age gate. v1 reads config from env at startup; v2 exposes a config-driven API (`/admin/photo-gc/*`) for a future UI. The sweep is guarded by a Postgres advisory lock so at most one sweep runs per cycle even under future multi-replica deploys.
- **Testcontainers integration for Garage** (`dxflrs/garage:v1.0.1`): a Tier-3 adapter test (PG + Garage) for the `PhotoStorage` adapter, and a Tier-4 round-trip test (PG + SierraDB + Garage) covering upload → saga → projection → download → unlink → refcount-driven deletion, plus the GC orphan sweep.

Not in scope: a `SeasonCrew` aggregate, API-driven GC config endpoints, presigned URLs for direct client<->Garage transfer, C2PA manifest handling (a spike to verify C2PA-stripping via re-encode is deferred), AI-image provenance preservation, `turbojpeg`/libheif native dependencies (pure-Rust `image` crate is the v1 choice), thumbnailing of non-image attachment types.

## Capabilities

### New Capabilities
- `photo-storage`: The `Photo` aggregate (SierraDB-tracked lifecycle), the `PhotoStorage` port (CRUD on bytes via OpenDAL/S3 against Garage), the upload/download API surface (proxy, authorisation-checked), the thumbnail-generation saga (`PhotoUploaded` → `OriginalNormalized` + `VariantGenerated`), the refcount-aware deletion saga chain (`PhotoUnlinked` → refcount → `DeletePhoto` → `PhotoBytesCleanup`), the periodic orphan GC task, and the Garage testcontainers harness.
- `photo-authorization`: The derived Season-scoped costume-photo authorisation policy (v1 Option A) plus the documented v2 evolution path to an additive `SeasonCrew` aggregate (Option B). Captures the authorisation rule, the between-blocks-gap limitation, and the non-breaking upgrade plan.

### Modified Capabilities
_(None — the existing `persistence-projections` requirements about projector count and event coverage will be extended in a backward-compatible manner by adding the Photo projector; no existing *requirement* is rewritten. The `costume-character-binding` capability is unchanged — `PhotoLinked`/`PhotoUnlinked` events on `Costume` are preserved as-is.)_

## Impact

- **`crates/core/src/photo/`** (new module) — `PhotoAggregate` (id, content_type, size_bytes, variants[], exif_stripped_at, deleted_at, version); commands (`UploadPhoto`, `NormalizeOriginal`, `GenerateVariant`, `MarkVariantFailed`, `DeletePhoto`); events (`PhotoUploaded`, `OriginalNormalized`, `VariantGenerated`, `VariantFailed`, `PhotoDeleted`); views (`PhotoView`, `PhotoVariantView`); ports (`PhotoStorage`, `PhotoCommands`, `PhotoRepository`); `PhotoId(Uuid)`, `PhotoVariant`, `VariantStatus`, `PhotoMetadata`, `PhotoBytes`, `PhotoGcConfig`, `SeasonPhotoAccessPolicy`.
- **`crates/core/src/membership/policy.rs`** (extended) — new `SeasonPhotoAccessPolicy` impl of `AuthorizationPolicy` deriving photo access from active costume-dept block membership in the season.
- **`crates/infra/src/photo/`** (new module) — `OpenDalPhotoStorage` (S3/Garage backend, key layout `{photo_id}/{variant}`), `PhotoProjector` (subscribes to the `photo` stream → `projection_photo`, `projection_photo_variant`), `PhotoThumbnailSaga` (reacts to `PhotoUploaded`; decodes with `image` + `kamadak-exif`; re-encodes original upright EXIF-stripped; generates thumb/medium; dispatches `NormalizeOriginal` + `GenerateVariant`), `PhotoDeletionSaga` (reacts to `PhotoUnlinked` on Costume; refcount check via `COUNT(*)` on `projection_costume_photo`; dispatches `DeletePhoto` when zero), `PhotoBytesCleanupSaga` (reacts to `PhotoDeleted`; `delete_all` from Garage; rows out of `projection_photo_*`), `PhotoGcSweepTask` (background tokio task; advisory-locked; env-configured; writes `projection_photo_gc_run` history), `PhotoCommandsImpl` + `PhotoRepositoryImpl` (kameo_es + sqlx).
- **`crates/infra/Cargo.toml`** — add `opendal` (features `services-s3`), `image` (features `jpeg`, `png`, `webp`), `kamadak-exif`.
- **`crates/infra/migrations/`** — new migration: `projection_photo` (photo_id PK, content_type, size_bytes, created_at, updated_at), `projection_photo_variant` (photo_id, variant, status, size_bytes, PK(photo_id, variant)), `projection_photo_gc_run` (run_id PK, started_at, finished_at, scanned, orphans_found, orphans_deleted, dry_run). `projection_costume_photo` already exists and is reused unchanged.
- **`crates/api`** — routes: `POST /costumes/{cid}/photos` (proxy upload, content-type allowlist, size cap, SeasonPhotoAccessPolicy), `GET /costumes/{cid}/photos/{pid}/bytes?variant=` (proxy download, authz per request, `Cache-Control: private, max-age=300`, streaming with ETag), `DELETE /costumes/{cid}/photos/{pid}` (SeasonPhotoAccessPolicy; dispatches `UnlinkPhoto`). `CostumePhotoView` enriched with `content_type`, `size_bytes`, `variants: Vec<PhotoVariantView>`. `ProductionPorts` + `Ports` trait grow a `PhotoStorage`, `PhotoCommands`, `PhotoRepo`, and the `SeasonPhotoAccessPolicy`.
- **`docker-compose.dev.yml` + `docker-compose.prod.yml`** — add a `garage` service (`dxflrs/garage:v1.0.1`) exposed only on the internal docker network (no host port mapping); env `S3_ENDPOINT`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_BUCKET=costume-photos`; persistent named volume.
- **`crates/integration-tests`** (ADR-014 one-harness rule) — Garage testcontainer helper (`dxflrs/garage:v1.0.1`); Tier-3 `PhotoStorage` adapter test; Tier-4 full round-trip (upload → saga → projection → download → unlink → refcount-driven deletion) + GC orphan sweep; excluded from `cargo-mutants`.
- **`backend/AGENTS.md`** — document the new env vars and the boot sequence (Garage must be up before the API: bucket creation, key provisioning).
- **New ADR** (ADR-019) — records the Garage-as-S3 decision (superseding/no-op vs ADR-009's Phase 1 `fs` plan), the Photo-as-aggregate decision (SSOT alignment, why not "photos as just storage"), the proxy-only serving decision (security over bandwidth), and the derived-authorisation v1 decision with the v2 SeasonCrew evolution pointer.
